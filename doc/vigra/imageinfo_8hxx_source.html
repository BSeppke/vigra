00170         /** Store image as given file type.
00171 
00172             This will override any type guessed
00173             from the file name's extension. Recognized file types:
00174 
00175             <DL>
00176             <DT><b>BMP:</b><DD> Microsoft Windows bitmap image file.
00177             <DT><b>EXR:</b><DD> OpenEXR high dynamic range image format.
00178             (only available if libopenexr is installed)
00179             <DT><b>GIF:</b><DD> CompuServe graphics interchange format; 8-bit color.
00180             <DT><b>HDR:</b><DD> Radiance RGBE high dynamic range image format.
00181             <DT><b>JPEG:</b><DD> Joint Photographic Experts Group JFIF format;
00182             compressed 24-bit color (only available if libjpeg is installed).
00183             <DT><b>PNG:</b><DD> Portable Network Graphic
00184             (only available if libpng is installed).
00185             <DT><b>PBM:</b><DD> Portable bitmap format (black and white).
00186             <DT><b>PGM:</b><DD> Portable graymap format (gray scale).
00187             <DT><b>PNM:</b><DD> Portable anymap.
00188             <DT><b>PPM:</b><DD> Portable pixmap format (color).
00189             <DT><b>SUN:</b><DD> SUN Rasterfile.
00190             <DT><b>TIFF:</b><DD> Tagged Image File Format.
00191             (only available if libtiff is installed.)
00192             <DT><b>VIFF:</b><DD> Khoros Visualization image file.
00193             </DL>
00194 
00195             With the exception of TIFF, VIFF, PNG, and PNM all file types store
00196             only 1 byte (gray scale and mapped RGB) or 3 bytes (RGB) per
00197             pixel.
00198 
00199             PNG can store UInt8 and UInt16 values, and supports 1 and 3 channel
00200             images. One additional alpha channel is also supported.
00201 
00202             PNM can store 1 and 3 channel images with UInt8, UInt16 and UInt32
00203             values in each channel.
00204 
00205             TIFF and VIFF are additionally able to store short and long
00206             integers (2 or 4 bytes) and real values (32 bit float and
00207             64 bit double) without conversion. So you will need to use
00208             TIFF or VIFF if you need to store images with high
00209             accuracy (the appropriate type to write is automatically
00210             derived from the image type to be exported). However, many
00211             other programs using TIFF (e.g. ImageMagick) have not
00212             implemented support for those pixel types.  So don't be
00213             surprised if the generated TIFF is not readable in some
00214             cases.  If this happens, export the image as 'unsigned
00215             char' or 'RGBValue<unsigned char>' by calling
00216             \ref ImageExportInfo::setPixelType().
00217 
00218             Support to reading and writing ICC color profiles is
00219             provided for TIFF, JPEG, and PNG images.
00220          **/

00224         /** Set compression type and quality.
00225 
00226             This option is ignored when the target file format doesn't recognize
00227             the compression type. Valid arguments:
00228 
00229             <DL>
00230             <DT><b>NONE:</b><DD> (recognized by EXR and TIFF): do not compress (many other formats don't
00231                            compress either, but it is not an option for them).
00232             <DT><b>JPEG:</b><DD> (recognized by JPEG and TIFF): use JPEG compression.
00233                            You can also specify a compression quality parameter by
00234                            passing "JPEG QUALITY=N", where "N" must be an integer between 1 and 100
00235                            (e.g. "JPEG QUALITY=70").
00236             <DT><b>JPEG-ARITH:</b><DD> (recognized by new versions of JPEG): use arithmetic coding as a back-end
00237                            after JPEG compression (by default, the back-end is Huffman coding).
00238                            You can also specify a compression quality parameter by
00239                            passing "JPEG-ARITH QUALITY=N", where "N" must be an integer between 1 and 100
00240                            (e.g. "JPEG-ARITH QUALITY=70").
00241             <DT><b>RLE, RunLength:</b><DD> (recognized by EXR and TIFF): use run-length encoding. (BMP also
00242                           uses run-length encoding, but there it is not an option).
00243             <DT><b>PACKBITS:</b><DD> (recognized by TIFF): use packbits encoding (a variant of RLE).
00244             <DT><b>DEFLATE:</b><DD> (recognized by TIFF): use deflate encoding, as defined in zlib (PNG also
00245                            uses deflate, but there it is not an option).
00246             <DT><b>LZW:</b><DD> (recognized by TIFF): use Lempel-Ziv-Welch encoding.
00247             <DT><b>ZIP:</b><DD> (recognized by EXR): use zip-style encoding.
00248             <DT><b>PIZ:</b><DD> (recognized by EXR): use wavelet encoding.
00249             <DT><b>PXR24:</b><DD> (recognized by EXR): reduce to 24-bit, then use zip-style encoding.
00250             <DT><b>B44, B44A:</b><DD> (recognized by EXR): see OpenEXR documentation.
00251             <DT><b>ASCII:</b><DD> (recognized by PNM): store pixels as ASCII (human readable numbers).
00252             <DT><b>RAW:</b><DD> (recognized by PNM): store pixels as uncompressed binary data.
00253             <DT><b>BILEVEL:</b><DD> (recognized by PNM): store as one bit per pixel.
00254             <DT><b>1 ... 100:</b><DD> deprecated (equivalent to <tt>setCompression("JPEG QUALITY=number")</tt>
00255                              where the number denotes the desired quality).
00256             </DL>
00257 
00258             Some of these options (e.g. "JPEG-ARITH", "LZW", "B44", "B44A") may only be available when
00259             they have been enabled in the corresponding third-party library.
